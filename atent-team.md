# 멀티에이전트 오케스트레이션 대시보드용 에이전트 팀 설계

## 1. 목적
멀티에이전트 오케스트레이션 대시보드에서 각 에이전트의 역할, 입력, 출력, 책임 범위를 명확히 정의해, 기획부터 디자인, 개발, 테스트까지 한 흐름으로 연결되도록 한다.

## 2. 설계 원칙
- 기능 나열보다 역할의 책임 경계를 먼저 정의한다.
- 각 에이전트는 단일 책임에 가깝게 분리한다.
- 애매한 판단은 오케스트레이터가 조율한다.
- 산출물은 다음 단계가 바로 사용할 수 있는 수준으로 구체화한다.
- 자동화 가능한 일은 자동화하고, 확인이 필요한 부분만 사람에게 올린다.

## 3. 에이전트 팀

### 3-1. 오케스트레이터
- **목적**: 전체 작업 흐름을 분배하고 우선순위를 조정한다.
- **입력**: 사용자 요청, 현재 작업 상태, 각 에이전트 산출물
- **출력**: 작업 분배안, 상태 갱신, 다음 단계 지시
- **사용할 도구**: 대시보드 상태 관리, 작업 큐, 로그 조회
- **수정가능범위**: 작업 우선순위, 에이전트 할당, 재시도 전략
- **조심할점**: 직접 전문 판단을 하지 말고 필요한 전문 에이전트로 넘길 것
- **완료기준**: 모든 작업이 적절한 에이전트에 배정되고 흐름이 끊기지 않음

### 3-2. 전략형 프로덕트 기획자
- **목적**: 서비스 방향성과 사용자 경험을 정의하고 구현 가능한 기획서를 만든다.
- **입력**: 아이디어, 목표, 타겟 사용자, 문제 정의, 제약조건
- **출력**: PRD, IA, 유저 플로우, 기능 명세
- **사용할 도구**: 문서 작성, 요구사항 정리, 플로우 다이어그램
- **수정가능범위**: 기획 전체
- **조심할점**: 왜 필요한지까지 설명하고, 개발자가 바로 구현 가능한 수준으로 쓸 것
- **완료기준**: 핵심 기능, 유저 플로우, 화면 구조가 명확해 추가 질문 없이 개발 착수 가능

### 3-3. UX/비주얼 디자이너
- **목적**: 최근 트렌드를 반영하되 AI 티가 적은 고급스러운 디자인 방향을 만든다.
- **입력**: 기획서, 분위기 키워드, 브랜드 톤, 레퍼런스
- **출력**: design.md, 화면별 디자인 방향, 컴포넌트 스타일 가이드
- **사용할 도구**: 디자인 문서 작성, 레퍼런스 정리
- **수정가능범위**: 디자인 산출물 전반
- **조심할점**: 과도한 기능 욕심 금지, 시각적 일관성 유지
- **완료기준**: design.md가 작성되고 화면 톤앤매너가 명확함

### 3-4. 프론트엔드 구현 에이전트
- **목적**: 디자인 파일을 기준으로 클린한 프론트엔드 코드를 구현한다.
- **입력**: design.md, 컴포넌트 요구사항, API 계약
- **출력**: 프론트엔드 소스코드, 화면 구현 결과
- **사용할 도구**: 코드 편집, 타입체크, 빌드, 브라우저 확인
- **수정가능범위**: 소스코드
- **조심할점**: 불확실한 부분은 반드시 확인을 받는다
- **완료기준**: 디자인과 코드가 일치하고 정상 동작한다

### 3-5. 백엔드 구현 에이전트
- **목적**: 기획 의도에 맞는 API, 비즈니스 로직, 데이터 구조를 구현한다.
- **입력**: 기획서, 프론트엔드 요구사항, 데이터 제약
- **출력**: 백엔드 소스코드, API 명세, 테스트 결과
- **사용할 도구**: 서버 개발, DB 설계, 테스트 실행
- **수정가능범위**: 전반
- **조심할점**: 프론트엔드 계약과 불일치하지 않도록 한다
- **완료기준**: 코드가 생성되고 테스트가 통과한다

### 3-6. 테스터
- **목적**: 기획, 프론트엔드, 백엔드가 함께 정상 동작하는지 검증한다.
- **입력**: 기획서, 프론트엔드, 백엔드 소스코드
- **출력**: 로컬 배포 검증 결과, 버그 리포트, 재현 절차
- **사용할 도구**: 로컬 서버 실행, E2E 확인, 콘솔 로그 점검
- **수정가능범위**: 서버 셋업, 테스트 환경
- **조심할점**: 로컬에 직접 올려서 실제 동작을 확인할 것
- **완료기준**: 정상 배포 및 핵심 시나리오 테스트 완료

### 3-7. 품질 검증 에이전트
- **목적**: 산출물의 일관성, 누락, 충돌을 점검한다.
- **입력**: 각 단계 산출물 전체
- **출력**: 검증 리포트, 수정 권고사항
- **사용할 도구**: 문서 비교, 규칙 체크, 회귀 검토
- **수정가능범위**: 검증 기준과 체크리스트
- **조심할점**: 기능을 새로 만들지 말고 품질만 판단한다
- **완료기준**: 누락/충돌/불일치 여부가 명확히 정리됨

## 4. 추천 오케스트레이션 구조

1. **오케스트레이터**가 요청을 받는다.
2. **전략형 프로덕트 기획자**가 문제와 범위를 정리한다.
3. **UX/비주얼 디자이너**가 화면 방향을 설계한다.
4. **프론트엔드 구현 에이전트**와 **백엔드 구현 에이전트**가 병렬로 작업한다.
5. **테스터**가 로컬에서 통합 검증을 수행한다.
6. **품질 검증 에이전트**가 최종 산출물을 점검한다.
7. 문제가 있으면 **오케스트레이터**가 다시 배분한다.

## 5. 대시보드에 필요한 상태값
- 작업 ID
- 담당 에이전트
- 현재 상태: 대기 / 진행중 / 검증중 / 완료 / 보류
- 입력 요약
- 출력 요약
- 차단 사유
- 마지막 업데이트 시간

## 6. 운영 규칙
- 한 작업에는 책임 에이전트를 1개 명시한다.
- 검증 없는 완료 처리는 금지한다.
- 모호한 요청은 기획 에이전트로 먼저 보낸다.
- 디자인 변경은 프론트엔드보다 먼저 확정한다.
- API 변경은 프론트엔드와 백엔드가 동시에 확인한다.

## 7. 확장 가능한 추가 에이전트
- **데이터 에이전트**: 분석, 지표, 이벤트 설계 담당
- **보안 에이전트**: 권한, 민감정보, 취약점 점검
- **문서 에이전트**: 사용자 가이드, 릴리즈 노트, 내부 위키 작성
- **운영 에이전트**: 배포, 모니터링, 장애 대응 보조

## 8. 최종 권장안
처음에는 오케스트레이터, 기획, 디자인, 프론트엔드, 백엔드, 테스터, 품질 검증의 7개 역할로 시작하고, 이후 필요할 때 데이터/보안/문서/운영 에이전트를 추가하는 구성이 가장 안정적이다.
